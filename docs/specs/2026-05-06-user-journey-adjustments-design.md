# Design Spec: การปรับระบบตาม User Journey ใบอนุญาต
**วันที่:** 2026-05-06  
**ที่มา:** สรุปจากการประชุมกับลูกค้า  
**Scope:** GCL - ระบบใบอนุญาตน้ำบาดาล

---

## ภาพรวม

จาก User Journey diagram "ใบอนุญาต" ที่คุยกับลูกค้า พบว่าต้องเพิ่ม/ปรับ 4 เรื่องหลัก แบ่งเป็น 2 Phase ตาม business priority:

- **Phase 1:** หน้าสมัครใช้งาน + Role เจ้าหน้าที่ภาคสนาม
- **Phase 2:** AI Recommendation + สร้างใบขวาง

---

## Phase 1

### 1. หน้าสมัครใช้งาน (Registration) — ผู้ประกอบการ

**ที่มา:** ปัจจุบันระบบยังไม่มีหน้าสมัครใช้งาน ผู้ประกอบการไม่สามารถ self-register ได้

**Scope:** ผู้ประกอบการ 3 ประเภทเท่านั้น (บุคคลธรรมดา, นิติบุคคล, วัด/โรงเรียน) — เจ้าหน้าที่ถูกสร้างโดย Admin

**Entry point:** เพิ่มปุ่ม "สมัครใช้งาน" ในหน้า Login (`01_ทั่วไป/01_หน้าเข้าสู่ระบบ.html`)

**Flow — 4-step wizard** (ใช้ pattern เดียวกับ ยื่นคำขอ Step 1–4):

| Step | ชื่อ | เนื้อหา |
|------|------|---------|
| 1 | เลือกประเภท | Card เลือก 1 อัน: บุคคลธรรมดา / นิติบุคคล / วัดหรือโรงเรียน |
| 2 | ข้อมูลส่วนตัว/องค์กร | ฟอร์มต่างกันตามประเภท (ดูรายละเอียดด้านล่าง) |
| 3 | ตั้งค่าบัญชี | email, password, เบอร์โทรศัพท์ |
| 4 | ยืนยัน | สรุปข้อมูล + ยืนยัน OTP/email |

**ฟอร์ม Step 2 แยกตามประเภท:**
- **บุคคลธรรมดา:** ชื่อ-นามสกุล, เลขบัตรประชาชน 13 หลัก, วันเกิด, ที่อยู่
- **นิติบุคคล:** ชื่อบริษัท, เลขนิติบุคคล 13 หลัก, ที่อยู่จดทะเบียน, ชื่อผู้มีอำนาจ
- **วัด/โรงเรียน:** ชื่อวัด/โรงเรียน, รหัสองค์กร (ถ้ามี), ที่อยู่, ชื่อผู้ดูแล

**Screens ที่ต้องสร้าง:**
```
html-screens/01_ทั่วไป/
  03_หน้าสมัครใช้งาน_Step1_เลือกประเภท.html
  04_หน้าสมัครใช้งาน_Step2_ข้อมูลส่วนตัว.html      ← แยก variant ตามประเภท
  05_หน้าสมัครใช้งาน_Step3_ตั้งค่าบัญชี.html
  06_หน้าสมัครใช้งาน_Step4_ยืนยัน.html
  07_หน้าสมัครสำเร็จ.html
```

### 2. Role ใหม่: เจ้าหน้าที่ภาคสนาม

**ที่มา:** User Journey มี actor "เจ้าหน้าที่ภาคสนาม" ทำหน้าที่ "ตรวจสถานที่&วัดพิกัด" แต่ระบบปัจจุบันยังไม่มี role หรือ screen รองรับเลย

**Screens ที่ต้องสร้าง (folder ใหม่):**
```
html-screens/05_เจ้าหน้าที่ภาคสนาม/
  01_หน้าหลัก.html
  02_บันทึกผลการตรวจสถานที่.html
  03_รายละเอียดผลที่ส่งแล้ว.html
```

#### หน้า 01 — หน้าหลัก
- แสดงรายการคำขอที่ได้รับมอบหมายให้ลงพื้นที่
- แต่ละ item แสดง: ชื่อผู้ขอ, ประเภทใบอนุญาต, สถานที่, วันที่ต้องตรวจ, สถานะ (รอลงพื้นที่ / ตรวจแล้ว)
- กด item → ไปหน้า 02

#### หน้า 02 — บันทึกผลการตรวจสถานที่
ฟอร์มประกอบด้วย:
- **พิกัด GPS:** ปุ่ม "ใช้ตำแหน่งปัจจุบัน" (auto-detect) หรือ pin บนแผนที่ด้วยตนเอง แสดง lat/lng
- **รูปถ่าย:** อัปโหลดได้หลายรูป (ต้องการอย่างน้อย 1 รูป)
- **ผลการตรวจ:** Radio — ✅ ผ่าน / ❌ ไม่ผ่าน
- **หมายเหตุ:** textarea (บังคับกรอกถ้าเลือก "ไม่ผ่าน")
- **ปุ่ม Submit:** ส่งผลกลับให้เจ้าหน้าที่หลัก → เปลี่ยนสถานะเป็น "ตรวจแล้ว"

#### หน้า 03 — รายละเอียดผลที่ส่งแล้ว
- Readonly view ของผลที่ submit แล้ว
- แสดง: พิกัด, รูปภาพ, ผลการตรวจ, หมายเหตุ, วันที่-เวลาที่ส่ง

#### ผลกระทบต่อ เจ้าหน้าที่หลัก
เพิ่ม **Tab ที่ 4 "ผลการตรวจสถานที่"** ในหน้าตรวจสอบคำขอ (ปัจจุบันมี 3 tab: เอกสาร / ฟอร์ม / แผนที่) แสดงข้อมูลที่ภาคสนามส่งมาแบบ readonly

```
html-screens/03_เจ้าหน้าที่/
  05b_ตรวจสอบคำขอ_แท็บผลตรวจสถานที่.html    ← ใหม่
```

## Phase 2

### 3. AI Recommendation — ประมวลผลข้อมูลช่วยตัดสินใจ

**ที่มา:** Step สีฟ้าใน User Journey diagram ระบบ AI วิเคราะห์คำขอและแนะนำผล

**ตำแหน่งใน UI:** Card แสดงในหน้าตรวจสอบคำขอของเจ้าหน้าที่ ด้านบนก่อน section ผลการตัดสิน

**เนื้อหาของ AI Card:**
- Badge ผลแนะนำ: ✅ ควรอนุมัติ / ❌ ควรปฏิเสธ + confidence level (เช่น 85%)
- รายการเหตุผลประกอบ (bullet) เช่น:
  - "พื้นที่มีบ่อเจาะแล้ว 3 บ่อในรัศมี 500 เมตร"
  - "ปริมาณน้ำในชั้นหินอุ้มน้ำอยู่ในเกณฑ์ปกติ"
  - "ไม่พบข้อขัดแย้งกับกฎระเบียบที่เกี่ยวข้อง"
- Disclaimer: "ผลนี้เป็นข้อมูลประกอบการตัดสินใจเท่านั้น เจ้าหน้าที่มีอำนาจในการตัดสินใจขั้นสุดท้าย"

**Screens:**
```
html-screens/03_เจ้าหน้าที่/
  03b_ตรวจสอบคำขอ_AI_Recommendation.html    ← variant ของหน้า 03 ที่มี AI Card
```

**หมายเหตุ:** AI Card ต้องรอ backend/data pipeline พร้อมก่อน — UI สามารถทำ mockup ได้ แต่ควร design ให้รองรับ loading state และ error state (กรณี AI ยังไม่พร้อม)

---

### 4. สร้างใบขวาง (Document สำหรับการประชุม)

**ที่มา:** Step "สร้างใบขวาง (สำหรับการประชุม)" ใน User Journey ก่อนนำเรื่องเสนอคณะกรรมการ

**Trigger:** ปุ่ม "สร้างใบขวาง" ในหน้าตรวจสอบคำขอของเจ้าหน้าที่ (หลังตัดสินใจเบื้องต้นแล้ว)

**Flow:**
1. กดปุ่ม → เปิด Modal preview ใบขวาง
2. ระบบ auto-fill ข้อมูลจากคำขอ (ชื่อผู้ขอ, ประเภท, สถานที่, ผลการตรวจ, ผล AI)
3. เจ้าหน้าที่แก้ไขเพิ่มเติมได้ก่อน generate
4. กด "Generate PDF" → ดาวน์โหลด PDF ใบขวาง

**Screens:**
```
html-screens/03_เจ้าหน้าที่/
  12_Modal_สร้างใบขวาง.html    ← ใหม่ (ใช้ pattern เดียวกับ Modal_ลงนามออกใบอนุญาต)
```

## Role Matrix (อัปเดต)

| Role | Phase | สถานะ |
|------|-------|--------|
| ผู้ประกอบการ | — | ✅ มีอยู่แล้ว (เพิ่มแค่ Registration) |
| เจ้าหน้าที่ | — | ✅ มีอยู่แล้ว (เพิ่ม Tab + ปุ่มใบขวาง) |
| เจ้าหน้าที่ภาคสนาม | Phase 1 | ❌ ต้องสร้างใหม่ทั้งหมด |
| คณะกรรมการส่วนกลาง/เขต | TBD | ⏳ ยังไม่ได้ตกลง scope |

## สิ่งที่ยังไม่อยู่ใน scope

- **Portal คณะกรรมการ** — TBD รอ confirm จากลูกค้าว่าต้องการ login เข้าระบบเองหรือให้เจ้าหน้าที่บันทึกแทน
- **Backend/API** — spec นี้ครอบคลุมเฉพาะ UI/UX mockup
- **Mobile App** — ระบบปัจจุบันเป็น Web-based, field officer อาจต้องการ mobile-friendly UI แต่ยังไม่ได้ระบุ

## Phase 3 (กำหนดทำในอนาคต)

**User Journey: เจาะ/หลังเจาะ** — flow ที่เกิดหลังจากผู้ประกอบการได้รับใบอนุญาตแล้ว

**Actors ใหม่ที่ต้องออกแบบ:**
- ช่างเจาะ/วิศวกร — เซ็นต์รับรองการเจาะ, ส่งรายงานผลชั้นดิน/หิน, บันทึกข้อมูลการเจาะ
- นักธรณีวิทยา — บันทึกข้อมูลทางวิชาการ

**Flow หลัก:**
- **เจาะ:** ได้รับใบอนุญาต → ดำเนินการเจาะ → Decision ความลึก > 300 ม. (แยก flow การรับรอง) → บันทึกข้อมูลการเจาะ
- **หลังเจาะ:** บันทึกข้อมูลหลังเจาะ (ก่อนใช้) → บันทึกข้อมูลบ่อ (Maintenance) → บันทึกข้อมูลทางวิชาการ

**หมายเหตุ:** เลื่อนไป Phase 3 เนื่องจาก role ใหม่ 2 ตัว + domain ใหม่ ควรทำ Phase 1-2 ให้เสร็จก่อน
